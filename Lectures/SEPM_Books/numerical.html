üéØ Overview: The Big Picture

What is Software Estimation?

Software estimation is the process of predicting how much time, effort, and money will be needed to build a software system. Before starting any project, clients and development teams need to answer two fundamental questions:

  • "How big is this project?" (Size Estimation)
  • "How much will it cost?" (Cost Estimation)
üí° Think of it Like Building a House

Before you build a house, you need to know:

  • Size: How many rooms? How many square feet? (Similar to counting features in software)
  • Cost: Based on the size, how much will materials and labor cost? (Similar to calculating development cost)

Just like you cannot price a house without knowing its size and features, you cannot estimate software cost without analyzing the requirements first!

üìö Three Main Estimation Methods Covered:

  • Function Point (FP) Analysis: Measures size based on functionality (what the software does)
  • COCOMO (Constructive Cost Model): Uses lines of code to estimate effort and time
  • COCOMO-II (Advanced Model): Modern approach that handles complex projects better

‚ö†Ô∏è Critical Rule: You CANNOT perform accurate estimation without first analyzing the problem and identifying the computing requirements. Without knowing what the system must do, any estimate is just guessing!

üéØ Why Software Estimation is Important

For Clients:

  • Budget planning and financial approval
  • Comparing bids from different vendors
  • Making informed business decisions

For Development Teams:

  • Resource allocation (how many developers needed?)
  • Project scheduling (when will it be done?)
  • Risk management (identifying potential problems early)
  • Setting realistic expectations

üìè Method 1: Function Point (FP) Analysis

What is Function Point Analysis?
Function Point (FP) Analysis is a method for measuring software size based on the functions it provides to users. The key advantage is that it is technology-independent ‚Äî it does not matter if you use Java, Python, or C++. It measures "functional size" based purely on requirements.
üè† House Analogy: When estimating a house's size, you do not measure by the type of bricks used. Instead, you count rooms, bathrooms, and features. Similarly, FP counts software features regardless of the programming language used.

The Five Components We Count

Function Point Analysis examines the software requirements and counts five specific components:

  1. External Inputs (EI): Data coming INTO the system from users or other systems.
    Example: A "Login" form where users enter username and password, or a "Register New User" screen.
  2. External Outputs (EO): Processed data going OUT of the system.
    Example: A "Monthly Sales Report" that shows calculated totals, or a "Receipt" after purchase.
  3. External Inquiries (EQ): User asks for data, system retrieves and displays it WITHOUT changing anything.
    Example: Searching for a book by title, or checking your bank account balance.
  4. Internal Logical Files (ILF): Data tables or files stored and maintained INSIDE the system.
    Example: A "Students" database table, or a "Products Inventory" file.
  5. External Interface Files (EIF): Data stored by ANOTHER system but used by OUR system.
    Example: Using Google Maps API for location data, or reading tax rates from a government database.
üí° Easy Way to Remember

EI (External Input): User GIVES data to the system ‚Üí "I want to register"

EO (External Output): System GIVES processed results ‚Üí "Here is your report"

EQ (External Inquiry): User ASKS, system SHOWS ‚Üí "Show me product details"

ILF (Internal Logical File): Data WE store ‚Üí "Our customer database"

EIF (External Interface File): Data OTHERS store ‚Üí "Data from payment gateway"

üìö Complete Example: Online Library System

Let us analyze a simple online library management system:

  • EI: "Add New Book" form, "Borrow Book" button ‚Üí 2 inputs
  • EO: "Overdue Books Report", "Receipt for borrowed book" ‚Üí 2 outputs
  • EQ: "Search book by author", "Check book availability" ‚Üí 2 inquiries
  • ILF: "Books table", "Members table", "Borrowing records table" ‚Üí 3 files
  • EIF: "Publisher information database" (external) ‚Üí 1 file

Next step: Classify each as Simple, Average, or Complex based on complexity.

Step-by-Step Calculation Method

Step 1: Calculate Unadjusted Function Points (UFP)

Each component has a weight based on its complexity. Here is the standard weight table:

Component Type Simple Average Complex
External Inputs (EI) 3 4 6
External Outputs (EO) 4 5 7
External Inquiries (EQ) 3 4 6
Internal Logical Files (ILF) 7 10 15
External Interface Files (EIF) 5 7 10
üî¢ Numerical Example: Student Registration System

After analyzing requirements, we identify:

  • EI: 2 simple login screens (2 √ó 3 = 6), 1 average "Add Course" form (1 √ó 4 = 4)
  • EO: 1 simple confirmation message (1 √ó 4 = 4), 1 complex "Transcript Report" (1 √ó 7 = 7)
  • EQ: 1 average "Search Course" function (1 √ó 4 = 4)
  • ILF: 1 complex "Students" table (1 √ó 15 = 15), 1 average "Courses" table (1 √ó 10 = 10)
  • EIF: 1 simple "Payment System" interface (1 √ó 5 = 5)

Calculate total UFP:

EI_total = 6 + 4 = 10

EO_total = 4 + 7 = 11

EQ_total = 4

ILF_total = 15 + 10 = 25

EIF_total = 5

UFP = 10 + 11 + 4 + 25 + 5 = 55 points

Step 2: Calculate Value Adjustment Factor (VAF)

The VAF (Value Adjustment Factor) adjusts our count based on 14 General System Characteristics (GSC). These come from non-functional requirements like performance, security, and usability.

üí° Why We Need VAF

Two systems might have the same number of functions, but one might be more complex to build due to:

  • High security requirements
  • Need to work on multiple platforms
  • Complex calculations
  • High performance needs

VAF accounts for these additional complexities!

The 14 General System Characteristics (GSC):
  • Data Communications (how much data exchange?)
  • Distributed Data Processing (multiple locations?)
  • Performance Requirements (speed critical?)
  • Heavily Used Configuration (system resources?)
  • Transaction Rate (how many operations per second?)
  • Online Data Entry (real-time input?)
  • End-User Efficiency (easy to use?)
  • Online Update (real-time database changes?)
  • Complex Processing (difficult calculations?)
  • Reusability (will code be reused?)
  • Installation Ease (easy to install?)
  • Operational Ease (easy to operate?)
  • Multiple Sites (deployed in many places?)
  • Facilitate Change (easy to modify?)

Rate each from 0 to 5: 0 = No Influence, 5 = Essential

VAF Calculation Formulas:

Step 1: Sum all 14 ratings to get TDI (Total Degree of Influence):

TDI = Sum of all 14 GSC ratings

Step 2: Calculate VAF:

VAF = (TDI √ó 0.01) + 0.65

üî¢ VAF Calculation Example

After reviewing our student system requirements, suppose we rated all 14 characteristics and got:

TDI = 40 (sum of all ratings)

VAF = (40 √ó 0.01) + 0.65

VAF = 0.40 + 0.65

VAF = 1.05

Note: VAF always ranges from 0.65 (minimum) to 1.35 (maximum)

Step 3: Calculate Final Function Points (FP)

Final FP Formula:

FP = UFP √ó VAF

üî¢ Final FP Calculation

FP = 55 √ó 1.05

FP = 57.75 Function Points

Result: The estimated size of our Student Registration System is 57.75 Function Points. This is our size metric that we can now use to calculate project cost!

üíµ Converting Function Points to Project Cost

Once we have the FP count, we need to convert it into actual money for bidding. There are three popular methods:

Method 1: Using Productivity Rate

Productivity Rate: This is the average number of Function Points one developer can complete in one month. It varies based on:
  • Programming language used
  • Team experience level
  • Project complexity
  • Development tools available

Typical range: 2 to 10 FP per Person-Month (average is about 5)

Cost Calculation Formulas:

effort_months = FP √∑ productivity_rate

development_cost = effort_months √ó monthly_labor_rate

total_cost = development_cost + overhead + contingency

bid_price = total_cost + profit_margin

üî¢ Complete Cost Example: Student System (57.75 FP)

Given Information:

  • Function Points = 57.75 FP
  • Productivity Rate = 5 FP per Person-Month
  • Monthly Labor Rate = 8,000 dollars per Person-Month
  • Overhead = 30 percent of development cost
  • Contingency Buffer = 10 percent for unexpected issues
  • Profit Margin = 15 percent

Step 1: Calculate Effort

effort_months = 57.75 √∑ 5 = 11.55 Person-Months

This means: 2 developers working for 5.8 months, OR 3 developers working for 3.85 months

Step 2: Calculate Development Cost

development_cost = 11.55 √ó 8,000

development_cost = 92,400 dollars

Step 3: Add Overhead

overhead_amount = 30 percent √ó 92,400 = 27,720 dollars

cost_with_overhead = 92,400 + 27,720 = 120,120 dollars

Step 4: Add Contingency

contingency_amount = 10 percent √ó 120,120 = 12,012 dollars

total_project_cost = 120,120 + 12,012 = 132,132 dollars

Step 5: Add Profit Margin

profit_amount = 15 percent √ó 132,132 = 19,820 dollars

Final Bid Price = 132,132 + 19,820 = 151,952 dollars

üìä Cost Breakdown:
‚Ä¢ Development: 92,400 dollars (60.8 percent)
‚Ä¢ Overhead: 27,720 dollars (18.2 percent)
‚Ä¢ Contingency: 12,012 dollars (7.9 percent)
‚Ä¢ Profit: 19,820 dollars (13.0 percent)
Total: 151,952 dollars

Method 2: Using Historical Cost per FP

Many companies keep records of past projects to know their average cost per Function Point.

Direct Cost Formula:

project_cost = FP √ó cost_per_FP

bid_price = project_cost √ó (1 + profit_percentage)

üî¢ Historical Data Method

Scenario: A software company has data showing their average cost is 2,500 dollars per Function Point.

Calculation:

project_cost = 57.75 √ó 2,500 = 144,375 dollars

bid_price = 144,375 √ó 1.15 = 166,031 dollars

With 15 percent profit, bid price = 166,031 dollars

Note: Cost per FP varies widely (500 to 5,000 dollars or more) depending on industry and complexity. Banking and healthcare systems typically cost more than simple business applications.

If you want to use COCOMO models (explained next), convert FP to Lines of Code using language-specific factors.

üí∞ Method 2: COCOMO (Constructive Cost Model)

What is COCOMO?
COCOMO (Constructive Cost Model) is a model for estimating software development effort and time. It uses the size of the project measured in KLOC (Kilo Lines of Code) which means thousands of lines of code.
üèóÔ∏è Construction Analogy: COCOMO is like estimating building time based on square footage. More square feet = more construction time. Similarly, more lines of code = more development time.

Three Project Modes in COCOMO

COCOMO categorizes projects into three modes based on complexity:

1. Organic Mode (Simple Projects)

Characteristics:

  • Small, experienced team
  • Simple, well-understood requirements
  • Familiar problem domain
  • Flexible requirements

Example: A basic data-entry application for company's internal use, or a simple inventory management system.

2. Semi-Detached Mode (Medium Projects)

Characteristics:

  • Medium-sized team with mixed experience
  • Medium complexity requirements
  • Some new features, some familiar
  • Moderately rigid requirements

Example: A university student portal with registration, grades, and announcements.

3. Embedded Mode (Complex Projects)

Characteristics:

  • Large team, possibly inexperienced with domain
  • Very complex, innovative requirements
  • Tight hardware constraints
  • Very rigid requirements

Example: Flight control software, medical device operating system, or missile guidance system.

COCOMO Calculation Formulas

Two Main Formulas:

Effort Formula:

effort_months = a √ó (KLOC to power b)

Time Formula:

development_time_months = c √ó (effort_months to power d)

Project Mode a b c d
Organic 2.4 1.05 2.5 0.38
Semi-Detached 3.0 1.12 2.5 0.35
Embedded 3.6 1.20 2.5 0.32
üî¢ Complete Example: Organic Project

Scenario: Developing a database utility tool. Team is experienced, requirements are clear. Project type = Organic. Estimated size = 10 KLOC (10,000 lines of code).

Step 1: Calculate Effort

Using Organic parameters: a = 2.4, b = 1.05

effort_months = 2.4 √ó (10 to power 1.05)

effort_months = 2.4 √ó 11.22

effort_months = 26.9 Person-Months

Step 2: Calculate Development Time

Using Organic parameters: c = 2.5, d = 0.38

development_time = 2.5 √ó (26.9 to power 0.38)

development_time = 2.5 √ó 3.53

development_time = 8.8 Months

Final Result: Project needs 26.9 Person-Months of effort and will take 8.8 months to complete.

Team Size: Average team size = 26.9 √∑ 8.8 = about 3 developers

üìä Example: Semi-Detached Project

Scenario: Building university registration portal. Medium complexity. Size = 20 KLOC

Effort Calculation:

effort_months = 3.0 √ó (20 to power 1.12)

effort_months = 3.0 √ó 28.5

effort_months = 85.5 Person-Months

Time Calculation:

development_time = 2.5 √ó (85.5 to power 0.35)

development_time = 12.6 Months

This means: Team of 7 developers (85.5 √∑ 12.6) working for 12.6 months

üè¶ Real-World Example: Banking System

A bank wants to build an ATM management system:

  • Complexity: Embedded mode (high security, real-time processing)
  • Estimated Size: 50 KLOC

Calculations:

effort_months = 3.6 √ó (50 to power 1.20) = 316 Person-Months

development_time = 2.5 √ó (316 to power 0.32) = 16.5 Months

Result: Needs team of 19 developers working for 16.5 months

üöÄ Method 3: COCOMO-II (Advanced Model)

What is COCOMO-II?
COCOMO-II is the improved successor to original COCOMO. It was developed because software development changed. The original model did not handle modern practices like object-oriented programming, code reuse, rapid prototyping, and component-based development.
üí° Why COCOMO-II is Better

Imagine you are estimating house construction:

  • Old Method (COCOMO): Just count total square feet
  • New Method (COCOMO-II): Consider if you are reusing prefab walls, if contractors are experienced, if location has good infrastructure, etc.

COCOMO-II accounts for MANY more factors that affect development time and cost!

Three Models for Different Project Stages

1. Application Composition Model

When to use: Very early stage, during prototyping

Size metric: Object Points (counts screens, reports, modules)

Best for: Systems built using existing components or frameworks

Example: Building a web app using drag-and-drop tools, or using WordPress for a website.

2. Early Design Model

When to use: After requirements are defined, before detailed architecture

Size metric: Function Points (FP) or KLOC

Best for: Getting budget estimates before detailed design

Example: After gathering all user requirements but before deciding technical architecture.

3. Post-Architecture Model (Most Detailed)

When to use: After system architecture is complete

Size metric: KLOC or Function Points

Special features: Uses 17 Cost Drivers and 5 Scale Factors for precision

Example: After deciding to use cloud database, microservices architecture, and specific frameworks.

Post-Architecture Model: Detailed Method

Main COCOMO-II Formula:

effort_months = A √ó (size to power B) √ó EAF

Where:

  • A = constant value, typically 2.94
  • size = project size in KLOC
  • B = exponent calculated from 5 Scale Factors
  • EAF = Effort Adjustment Factor (from 17 Cost Drivers)

Step 1: Calculate Exponent B Using Scale Factors

Rate these 5 Scale Factors from Very Low (5) to Extra High (0):

Scale Factor What it Measures Rating Range
PREC (Precedentedness) How familiar is this type of project? 5 (very new) to 0 (done many times)
FLEX (Development Flexibility) Can requirements change easily? 5 (very rigid) to 0 (very flexible)
RESL (Risk Resolution) Have major risks been addressed? 5 (many unknowns) to 0 (all resolved)
TEAM (Team Cohesion) How well does team work together? 5 (poor teamwork) to 0 (excellent teamwork)
PMAT (Process Maturity) How mature is development process? 5 (no process) to 0 (highly mature)

Formula for Exponent B:

B = 0.91 + (0.01 √ó sum_of_all_5_scale_factors)

üî¢ Example: Calculating B

For our project, we rate:

  • PREC = 3 (somewhat familiar project type)
  • FLEX = 2 (moderate flexibility in requirements)
  • RESL = 3 (some risks still present)
  • TEAM = 1 (good team cohesion)
  • PMAT = 2 (decent process maturity)

sum_of_scale_factors = 3 + 2 + 3 + 1 + 2 = 11

B = 0.91 + (0.01 √ó 11)

B = 0.91 + 0.11

B = 1.02

Step 2: Calculate EAF from Cost Drivers

There are 17 Cost Drivers grouped into four categories. Each is rated from Very Low to Extra High. Nominal (average) always equals 1.00.

Cost Driver What it Measures Very Low Nominal Extra High
Product Attributes
RELY Required Reliability 0.82 1.00 1.26
DATA Database Size 0.90 1.00 1.14
CPLX Product Complexity 0.73 1.00 1.74
Platform Attributes
TIME Execution Time Constraint - 1.00 1.63
STOR Main Storage Constraint - 1.00 1.57
Personnel Attributes
ACAP Analyst Capability 1.42 1.00 0.71
PCAP Programmer Capability 1.34 1.00 0.76
PEXP Platform Experience 1.19 1.00 0.85
Project Attributes

SITE Multi-Site Development 1.22 1.00 0.83
SCED Required Schedule 1.43 1.00 1.00

Note: Showing 11 of 17 drivers to illustrate concept

üí° Understanding Cost Drivers

How to read the table:

  • Values less than 1.00: Reduce effort (good factors like experienced team)
  • Values greater than 1.00: Increase effort (challenging factors like high complexity)
  • Value equals 1.00: Average, no adjustment needed

Formula for EAF:

EAF = multiply all 17 cost driver values together

üî¢ Example: Calculating EAF

After analyzing our project:

  • High Reliability needed: RELY = 1.10
  • High Analyst Capability: ACAP = 0.86
  • Good Development Tools: TOOL = 0.90
  • All other 14 drivers are Nominal = 1.00

EAF = 1.10 √ó 0.86 √ó 0.90 √ó (1.00 fourteen times)

EAF = 0.85

The good team and tools offset the high reliability requirement!

Step 3: Calculate Final Effort

üî¢ Complete COCOMO-II Calculation

Given values:

  • A = 2.94 (constant)
  • size = 30 KLOC
  • B = 1.02 (from Scale Factors)
  • EAF = 0.85 (from Cost Drivers)

Calculate Effort:

effort_months = 2.94 √ó (30 to power 1.02) √ó 0.85

effort_months = 2.94 √ó 31.76 √ó 0.85

effort_months = 79.5 Person-Months

Why COCOMO-II is More Accurate: This estimate considers team skills, project complexity, tools available, and many other factors that all come from analyzing requirements. It gives a much more realistic estimate than simple line counting!

üõí Real-World Application: E-Commerce Platform

An online shopping company is building a new platform:

Stage 1: Application Composition Model

Early prototype using existing UI components. Estimate time for building all screens and basic features.

Stage 2: Early Design Model

After defining all requirements (search, cart, payment, reviews), use Function Points to get initial budget estimate.

Stage 3: Post-Architecture Model

After deciding on cloud architecture and discovering they can reuse 30 percent of code from old system, this model allows inputting code reuse percentage, which correctly lowers the final estimate.

Key Benefit: COCOMO-II adapts as you learn more about the project, giving increasingly accurate estimates at each stage!

üìö Summary and Key Takeaways

Function Point (FP) Analysis

  • Measures functional size based on what users can do
  • Technology-independent (works for any programming language)
  • Counts 5 components: EI, EO, EQ, ILF, EIF
  • Adjusts for complexity using 14 General System Characteristics
  • Final formula: FP = UFP √ó VAF
  • Best for: Early estimation, comparing different technologies

COCOMO (Constructive Cost Model)

  • Uses KLOC (Kilo Lines of Code) as size input
  • Three modes: Organic (simple), Semi-Detached (medium), Embedded (complex)
  • Provides both effort and time estimates
  • Simple to use but less accurate for modern development
  • Formulas: effort = a √ó (KLOC to power b) and time = c √ó (effort to power d)
  • Best for: Quick estimates when you know lines of code

COCOMO-II (Advanced Model)

  • Three models for different project stages (prototype, early design, detailed design)
  • Handles modern practices: object-oriented, code reuse, components
  • Uses 5 Scale Factors to calculate exponent B
  • Uses 17 Cost Drivers to calculate EAF (Effort Adjustment Factor)
  • Much more accurate and flexible than original COCOMO
  • Formula: effort = A √ó (size to power B) √ó EAF
  • Best for: Large, complex projects with detailed requirements

üéØ Which Method to Use?

  • Use FP Analysis when: You have clear requirements but no code yet, or comparing different technology options
  • Use COCOMO when: You need a quick estimate and can estimate lines of code, and project is straightforward
  • Use COCOMO-II when: Project is complex, you have detailed information about team and tools, or you need the most accurate estimate possible
  • Best Practice: Use multiple methods and compare results for most reliable estimates!
